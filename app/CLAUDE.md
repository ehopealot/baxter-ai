# The mail agent ("Baxter Burgundy")

A daemon (`scripts/poll.mjs`) polls a dedicated AgentMail inbox every `POLL_INTERVAL_SECONDS`. For each new, allowlisted, non-automated thread, it spawns one scoped `claude -p` run (the persona "Baxter Burgundy") with a rendered prompt containing the full thread transcript. That run can reply in-thread, research the web via `WebSearch`/`WebFetch` and browse/interact via the Playwright CLI skill (persistent browser profile → logins/cookies carry over between emails), and read/write a small cross-thread memory file. Configured via `app/.env` (copy from `app/.env.example` — every var is commented there).

Run it via the root `Makefile`: `make run` starts the **default fleet** (Discord gateway + heartbeat + codapi) detached, docker-compose-managed (`compose.yaml`). **The mail poller (`poll.mjs`) is opt-in** — `make run-mail` adds it (gated behind compose's `mail` profile; opt-in only because Discord is the default surface, not for any maintenance reason — AgentMail is a single API key). `make mail`/`make discord` run a single surface in the foreground, `make stop` tears the fleet down, `make inbox` (one-time AgentMail inbox provisioning), `make app-shell` (debug shell). See root `CLAUDE.md` for the compose model (images built by the Makefile; network + config volume are `external`). 

## File map

- **`scripts/poll.mjs`** — the daemon loop. Groups new messages by thread, dispatches at most one `claude -p` run per thread per cycle, enforces `MAX_EMAILS_PER_CYCLE`/`MAX_SENDS_PER_DAY`, and writes the 0600 AgentMail API key file at startup (see Auth below).
- **`scripts/mail.mjs`** — the credential boundary for the mail surface (reads the AgentMail API key env-first-then-file). CLI wrapper around the AgentMail API via the `agentmail` SDK (`list-new`, `get-thread`, `reply`, `send`, `label`), invoked as a subprocess by both `poll.mjs` and the spawned `claude -p` run's `Bash` tool. `list-new` pages the inbox from a persisted timestamp cursor (`MAIL_POLL_CURSOR_PATH`, held one margin below the oldest unhandled message) and excludes the `agent-processed` label (the exactly-once source of truth) plus own mail; every `reply`/`send` applies a `baxter-sent` label so the agent's own messages are self-identifiable. The pure cores (`classifyListing`/`buildThreadOutput`/…) are unit-tested with an injected fake client, and the SDK is imported lazily so the tests need no network. The CLI-dispatch block is guarded with an `import.meta.url`/`pathToFileURL` check so importing the pure exports doesn't trigger dispatch.
- **`scripts/transcript.mjs`** — the provider-neutral transcript-forgery sanitization pipeline (`normalizeTranscriptText`/`neutralizeStructuralMarkers`/`neutralizeDanglingSeparatorTail`/`formatThreadMessage`/`extractEmailAddress` + `TRIGGER_MARKER`/`MESSAGE_SEPARATOR`), extracted out of the old `gmail.mjs` and shared by `mail.mjs` and `discord-bot.mjs`. `formatThreadMessage` takes a provider-neutral normalized message `{from, date, subject, text, isOwn, isAllowed}`. See the sanitization-pipeline section below.
- **`scripts/paths.mjs`** — all persistent state paths (`~/.mail-agent/...`, i.e. the config Docker volume), centralized so they can't drift. Includes `AGENTMAIL_KEY_PATH` (the 0600 API key file) and `MAIL_POLL_CURSOR_PATH` (the poll cursor); `MEMORY_PATH` is deliberately isolated in its own `memory-workspace/` subdirectory — see Sandbox below.
- **`scripts/send-state.mjs`** — the daily send-cap counter (`MAX_SENDS_PER_DAY`), incremented by `mail.mjs` at the actual send call (not by `poll.mjs` at dispatch time — a run can send more than one email).
- **`scripts/make-inbox.mjs`** — one-time AgentMail inbox provisioning (`make inbox`): create-or-show Baxter's inbox and print `AGENTMAIL_INBOX_ID`/`BAXTER_EMAIL` for `app/.env`.
- **`scripts/invisible_cli.py`** — the stealth browser wrapper (installed on PATH as `invisible-cli`). See "Two browsers" below.
- **`skills/invisible-playwright/`** — the skill documenting `invisible-cli`, dropped into the run's cwd by `poll.mjs` so it's discoverable. See "Two browsers".
- **`prompt.md`** — the template rendered per thread. Uses `{{FROM}}`/`{{SUBJECT}}`/`{{BODY}}`/`{{MESSAGE_ID}}`/`{{MEMORY_PATH}}`/`{{CREDENTIALS_PATH}}`/`{{MAIL_CLI_PATH}}`/`{{PERSONA_NAME}}`/`{{BAXTER_EMAIL}}`/`{{PROJECTS_LIST}}`/`{{LEARNED_SKILLS_LIST}}`/`{{LOADED_SKILLS}}` placeholders.

## Two browsers: playwright-cli (default) + invisible-cli (stealth)

Baxter has two independent browser paths, both exposed to the run via `Bash(...)` allow-rules:

- **`playwright-cli`** (Chromium, Node) — the default for ordinary browsing. Fast to start. Its skill is generated at build by `playwright-cli install --skills` under `/app/.claude/skills/playwright-cli`.
- **`invisible-cli`** (`scripts/invisible_cli.py`, Python) — an **anti-detect** browser for sites that fingerprint/block automation (Cloudflare "Just a moment…", bot walls). Backed by [`invisible_playwright`](https://github.com/feder-cr/invisible_playwright)'s patched Firefox, whose fingerprint (navigator/GPU/canvas/fonts/audio/WebRTC/timezone) is masked at the engine level — it presents as a **Windows desktop** even though the container is Linux/arm64, and `navigator.webdriver` is `false`.

**Why invisible-cli is a separate tool, not a browser flag on playwright-cli:** the patched Firefox speaks the Juggler protocol from Playwright 1.55 only (its `pyproject.toml` pins `playwright>=1.55,<1.56`; 1.61+ breaks it), while `@playwright/cli` bundles playwright-core 1.62 — so playwright-cli literally cannot drive this browser. The stealth also depends on invisible_playwright's own Python launcher (Firefox prefs + per-seed fingerprint), not just the binary. Hence a self-contained Python venv (`/opt/venv`) and a from-scratch CLI.

**invisible-cli design notes** (all learned by testing, see the wrapper's comments):
- Persistent-session **daemon + client** over a unix socket (`/tmp/invisible-cli.sock`), mirroring playwright-cli's model: `open` starts the daemon+browser, later commands reuse it, `close` ends it. Command surface mirrors playwright-cli (open/goto/snapshot/find/click/dblclick/fill/type/press/hover/select/check/uncheck/go-back/go-forward/reload/screenshot/eval/close).
- **Element refs are identical to playwright-cli**: `snapshot` calls Playwright's internal `snapshotForAI` (via the impl channel — its `send(method, timeout_calculator, params)` signature is frozen by the 1.55 pin) to get `[ref=eN]` handles, resolved with `aria-ref=eN` locators.
- **No persistent Firefox profile** — the patched build reliably launches a fresh profile but crashes on the *second* launch of a populated persistent profile ("Connection closed while reading from the driver"). Instead logins persist via a `storage_state` JSON file (`~/.mail-agent/invisible-state.json` on the config volume), saved after every page-changing command and on `close`, reloaded into a fresh stealth context — the same approach as playwright-cli's `state-save`/`state-load`.
- **"Headless" is headed-under-Xvfb** (invisible_playwright spawns its own Xvfb) because true headless is itself a detection signal — hence the `xvfb` package.
- The patched Firefox (~100 MB, arm64 build published upstream) is fetched at **build** time into `/opt/invisible-cache` (`XDG_CACHE_HOME`), NOT `/home/node` — the config volume shadows `/home/node` at runtime (same reason `PLAYWRIGHT_BROWSERS_PATH` points at `/opt`).
- Fixed fingerprint **seed** + pinned locale/timezone (`en-US`/`America/Los_Angeles`, env-overridable) so Baxter presents a consistent US device without a per-launch geoip lookup.
- Only one daemon runs at a time (single socket). The `close` handler unlinks the socket *before* the seconds-long browser teardown, so a new client can't connect to the dying daemon mid-shutdown (which would hit a closing page and error) — it finds no socket and spawns a fresh daemon.
- **Hang self-recovery.** Without it, a stuck browser blocked the client until the *outer* run-harness timeout (`OPENROUTER/OPENAI_CLI_TIMEOUT_MS`, ~120s). Two distinct hang shapes, both covered: a **page that never settles** hangs the client's `recv` (bounded by `INVISIBLE_CLI_CMD_TIMEOUT`, 30s — must exceed the daemon's own 20s snapshot timeout), while a **wedged Firefox launch** hangs *before* the socket exists (the daemon binds `SOCK_PATH` only after `start_browser()`), so it's caught in `_connect` (bounded by `INVISIBLE_CLI_CONNECT_TIMEOUT`, 30s). On either, the client SIGKILLs the daemon's whole **process group** (the daemon is a session leader via `start_new_session`, so its PID — written to `/tmp/invisible-cli.pid` at startup — == its pgid, taking Firefox + Xvfb down with it) — and it kills on *every* failed post-spawn connect, not just a `recv` timeout, so a wedged launch can't leak its browser stack. Then it unlinks the socket + pidfile; for `open` it respawns fresh and retries once (the "close and reopen"). Non-`open` commands have no session to resume and return an error telling the run to `open` again. Two sharp edges the pidfile mechanism has to survive: (1) `_shutdown` only unlinks the pidfile if it *still names the departing daemon* — a fast `close`→`open` means the new daemon may already have written its own PID there, and a blind unlink would silently disable recovery for the new session; (2) the whole invocation runs under a single monotonic budget (`INVISIBLE_CLI_BUDGET`, 110s — every blocking leg capped at `min(own cap, budget left)`) so even the worst compose of hang-legs stays under the 120s outer kill and its own cleanup + error always run.

`ensureSkills()` (in `runtime.mjs`, called by all three claude-spawning daemons) copies the baked skills (one base `SKILL_NAMES` list in `grants.mjs`, appended to by `make add-skill`; the build-generated `playwright-cli` one under `/app/.claude/skills` and the repo's `invisible-playwright`/`discord`/`code`/`web`/`projects`/`data`/`skill-discovery` ones under `/app/skills`. Each surface derives its staged set by **filtering the base list by its own exclusions** (`skillsExcept(...)` → `MAIL_/DISCORD_/HEARTBEAT_SKILL_SRCS`): mail drops `discord` (no `discord-cli` tool), heartbeat drops `schedule` (no `schedule-cli` — a fired task can't schedule), discord/voice drop nothing. The exclusions are the only skill asymmetries and each mirrors a missing tool; the tool grants, not the docs, are the enforced boundary) into `MEMORY_DIR/.claude/skills/` each run (skills resolve from cwd; refreshed so a run's unscoped `Write` can't leave them corrupt), and **also stages the agent's own learned skills** (see below). The run's `--allowedTools` grants `Bash(playwright-cli *)`, `Bash(invisible-cli *)`, `Bash(code-cli *)`, `Bash(web-cli *)` (keyless web fetch; its `search` subcommand is disabled — DDG blocks it, so it redirects to searching Bing in the browser; `scripts/web-cli.mjs` + the `web` skill — reaches the net from the daemon, no JS, falls back to the browsers when a page is JS-heavy), `WebSearch`/`WebFetch` (native web research; no new exfil surface — the browser CLIs already reach any URL), plus `Skill` (so the run can load a skill's full command reference, not just its one-line description).

**Learned skills — why the agent authors them outside `.claude/`:** Claude Code guards its own `.claude/` directory against writes by the spawned run (confirmed by testing: a `Write`/`Edit`, and even a Bash `>>` redirect, to `MEMORY_DIR/.claude/skills/...` is refused as an ungranted permission, while writes elsewhere in the cwd like `memory.md` succeed). So the run can't write into `.claude/skills` where its old design told it to. Instead it authors skills under `LEARNED_SKILLS_DIR` (`MEMORY_DIR/learned-skills/`, a plain dir in its writable cwd), and `ensureSkills` — running in the trusted Node daemon, which has no such guard — copies each learned-skill subdir into `.claude/skills` at the start of every run. So a skill the agent writes is discoverable on its **next** run, and (since `LEARNED_SKILLS_DIR` is under the shared `MEMORY_DIR`) is shared across the email and Discord surfaces. Two guardrails, since the run controls `learned-skills/` and its inputs are attacker-influenced: (1) a learned skill can **never shadow a baked one** — names in `BAKED_SKILL_NAMES` (`playwright-cli`/`invisible-playwright`/`discord`/`code`/`schedule`/`web`/`projects`/`data`/`skill-discovery`) are refused, so a poisoned `learned-skills/playwright-cli` can't overwrite the real skill; (2) staging is a **sync**, not accretion — `learned-skills/` is the source of truth, so removing a rogue/stale skill there (and it's gone from `.claude/skills` on the next run) is all the cleanup needed.

## Discord bot (second surface, same agent)

Baxter also lives in Discord. It's a **separate daemon from the mail poller**, built into the same image and sharing the same config volume, so memory, skills, and per-channel state carry across both surfaces. Run it with `make discord` (its own foreground container); `make run` starts the default detached fleet **including** the Discord gateway (plus heartbeat + codapi), while `make run-mail` also adds the (opt-in) mail poller. Unset `DISCORD_BOT_TOKEN` disables it (exits 0).

- **`scripts/discord-bot.mjs`** — the gateway daemon (discord.js v14). Holds the persistent websocket, and per `messageCreate` decides whether to respond, then spawns one scoped `claude -p` run per trigger via the shared `runtime.mjs` `runAgent` (same stream-json logging + out-of-tokens handling as email). `runAgent` is the harness-agnostic driver; the Claude Code specifics (invocation, stream-json decoding, rate-limit detection) live in `scripts/harnesses/claude.mjs`, selected via `BAXTER_HARNESS` (default `claude`; an unknown value fails at daemon startup). The two Claude-Code-isms it deliberately leaves caller-side — `allowedTools` (the enforced tool-permission boundary) and `.claude/skills` staging — are documented at the top of that adapter. **Two more harnesses** drive Baxter through a **structured-tool** layer instead of Claude Code (no shell: `run_cli` reinterprets `allowedTools`' `Bash(...)` patterns as an execFile allowlist, and read/write/edit are cwd-confined, writes also refusing `.claude/`; the executors live in `harnesses/openrouter-tools.mjs`, the shared preamble + tool set in `harnesses/runner-common.mjs`, and the runner→adapter event decoding in `harnesses/runner-events.mjs`): **`openrouter`** (`openrouter.mjs` + `openrouter-runner.mjs`) via OpenRouter's Agent SDK (`@openrouter/agent`), needs `OPENROUTER_API_KEY`/`OPENROUTER_MODEL`; and **`local`** (`local.mjs` + `local-runner.mjs`) driving any OpenAI **chat/completions** endpoint (self-hosted via Ollama/LM Studio, default `http://localhost:11434/v1`, or OpenAI/OpenRouter), needs `OPENAI_MODEL` (+ optional `OPENAI_BASE_URL`/`OPENAI_API_KEY`). Both experimental; the model must support tool calling. **Context-blowout guards** (these hand-rolled loops don't self-compact the way `claude -p` does): every tool result is size-capped — CLI output at `*_CLI_OUTPUT_MAX_BYTES`, and native `read`/`load_skill` too (previously the one uncapped path, `capRead` in `openrouter-tools.mjs`); the **local** runner also holds the whole message array under `OPENAI_CONTEXT_MAX_TOKENS` by stubbing the oldest tool results (then oversized tool-call arguments, e.g. a big `write_file`) in place before each call (`fitContext` in `runner-common.mjs` — keeps the system + original prompt and the tool_call/tool_result pairing intact, trims-and-continues); the **openrouter** runner can't trim mid-loop (the SDK owns the array) so it instead *stops* early via the SDK's `maxTokensUsed(OPENROUTER_MAX_TOKENS)` stop condition + `allowFinalResponse`. **If the window is exceeded anyway**, both runners recover rather than hard-fail: a context-full error is detected (`isContextFullError`) and the run trims + retries — the local runner halves its own array and retries (`OPENAI_CONTEXT_RETRY_MAX`); the openrouter runner truncates the oldest tool OUTPUTS in the SDK's saved `ConversationState` (`trimStateToolOutputs`) and resumes `callModel` (`OPENROUTER_CONTEXT_RETRY_MAX`) — this resume-after-overflow path is **verified live against minimax's 204k window**. **Last resort — model escalation:** any request failure that ISN'T out-of-credits/rate-limit — crucially INCLUDING minimax's opaque `invalid_prompt`/"invalid request" for an over-long request, which `isContextFullError` can't recognize (that miss once turned a ~1 MB scoreboard payload into a hard fail with the reply dropped) — triggers a one-shot resume on a larger-context fallback model (`shouldEscalateModel`; `OPENROUTER_FALLBACK_MODEL`, default the already-configured `OPENROUTER_MULTIMODAL_MODEL` = minimax-m3, ~1M window vs m2.7's ~205k). So a big tool payload becomes survivable (the bigger window swallows it, nothing trimmed) instead of a dropped reply; escalation is one-shot, skipped when already on the fallback, and reuses the saved `ConversationState` (or re-sends the original task if the SDK failed before saving). A shared `tryEscalate` closure applies it in BOTH the main tool loop AND the give-up **nudge/poke** resume, because an overflow can first surface at the nudge — the main loop fitting under the window but the poke's added turn tipping past it silently dropped an owed reply (2026-07-20) until the nudge path shared the escalation. Complements keeping responses small at the source (scoped queries). If trimming AND escalation still can't fit it, a **recognized** context-full error ends the run *gracefully* (exit 0, a clear "context full" result) so heartbeat doesn't retry into the same wall and discord/poll don't count it a hard failure; the opaque `invalid_prompt` shape (unrecognized by `isContextFullError`) still **hard-fails** (exit 1) when escalation is spent/disabled — a fresh heartbeat retry escalates again. See `docs/superpowers/specs/2026-07-17-openrouter-harness-design.md`. Reads `DISCORD_BOT_TOKEN`; the spawned run does not. **Multimodal routing (openrouter harness):** when `OPENROUTER_MULTIMODAL_MODEL` is set and a Discord trigger post carries media (image/video/audio/PDF), that one run is routed to that model (e.g. `minimax/minimax-m3`) with the attachment(s) attached to the first turn — the daemon detects media at notify-time (`selectMediaAttachments`, host-validated to the Discord CDN + type/count capped), carries it across the debounce coalesce (`_coalesce` unions prev+next media oldest-first so "image then caption" keeps the image), and passes it to the run via `BAXTER_MODEL_OVERRIDE` + `BAXTER_MEDIA`; the runner turns `BAXTER_MEDIA` into the SDK's `input_image`/`input_video`/`input_file`/`input_audio` content parts (`buildMediaParts`, camelCase — `callModel` serializes through the outbound schema; audio is base64, the rest are URL-passthrough). Empty knob or a media-less post → the default text model, unchanged. Independently of routing, every attachment is surfaced in the transcript as a short sanitized marker (`[image: cat.png]`, via `attachmentMarkers` in `renderHistory`) so even a text-only run — the model isn't set, or the type doesn't forward — knows media was posted instead of seeing an empty body. See `docs/superpowers/specs/2026-07-18-discord-multimodal-m3-design.md`.
- **`scripts/discord-cli.mjs`** — the token-scoped Discord REST CLI (raw `fetch` to API v10, no discord.js). The **only** thing besides the gateway that touches the token; the run reaches Discord only through `Bash(discord-cli *)`, mirroring how `mail.mjs` is the credential boundary for email. The boundary is real, not just prompt text: `discord-bot.mjs` writes the token to a 0600 file (`DISCORD_TOKEN_PATH`, outside the run's cwd) at startup and passes `runAgent` an env with `DISCORD_BOT_TOKEN` **stripped**, so the run can't echo the token out via an allowed `discord-cli` command — `discord-cli` reads it from the file (env fallback for direct use). Residual: the run's unscoped `Read` can still open the token file itself (0600 doesn't stop the same UID) — the same accepted exposure as `agentmail-key.json`; the env-strip closes the incidental echo-via-command path, not file access. Installed on PATH as `discord-cli` (Dockerfile shim). Command surface: `whoami`, `send`, `reply`, `dm` (gated — see below), `react`, `fetch-history`, `list-channels` (read-only discovery), `create-thread`, `send-thread`, `edit`, `delete-own` (ownership-enforced in code), `delete-any`, `pin`/`unpin`, `typing`. `dm <userId>` opens/reuses a private DM channel with any user sharing a guild and posts there (counts against the send cap) — a *wider reach* than a channel post (private + unaudited + any user), so it's **refused unless `DISCORD_ALLOW_DM=1` is in the env**, which is set in exactly one place: the voice bot's own `dmSpeaker()` when it deterministically DMs a voice requester the dispatch result. **No spawned run sets it** — the mail/discord/heartbeat/voice-dispatch runs can't DM at all (their inputs are attacker-influenced), so a prompt-injected run can't privately message arbitrary users; the DM is a code-owned voice-bot action to the person who just spoke, not a model-driven one. `delete-any` is the one moderation action that reaches other users' content: it deliberately has **no** code author-check (unlike `delete-own`) and is gated instead by Discord's own per-channel **Manage Messages** permission — the operator grants it only in the few channels where Baxter may moderate, and the DELETE of *another user's* message 403s everywhere else (deleting its own message needs no permission and works anywhere — that's `delete-own`), so the server, not prompt text, bounds where it can reach other people's messages. Editing others' messages is intentionally absent because Discord forbids it (only a message's author can edit its content), so `edit` stays own-messages-only. **No membership/role/channel-management command exists** — defense in depth beside the denied permissions. (`list-channels` is read-only channel *discovery* — `GET /users/@me/guilds` + `GET /guilds/{id}/channels` so a run can resolve a channel *name* to its id (positionals are name substrings, not guild ids), which it couldn't do before; it's not channel *management*: nothing creates, edits, reorders, or deletes channels.)
- **`discord-prompt.md`** — the per-message template (placeholders `{{PERSONA_NAME}}`/`{{BOT_USER}}`/`{{CHANNEL_ID}}`/`{{CHANNEL_KIND}}`/`{{SELF_ID}}`/`{{TRIGGER_AUTHOR}}`/`{{TRIGGER_MESSAGE_ID}}`/`{{HISTORY}}`/`{{MEMORY_PATH}}`/`{{CREDENTIALS_PATH}}`/`{{LEARNED_SKILLS_DIR}}`/`{{CHANNEL_MEMORY_PATH}}`/`{{PROJECTS_LIST}}`/`{{LEARNED_SKILLS_LIST}}`/`{{LOADED_SKILLS}}`).
- **`discord-reaction-prompt.md`** — the lean template for a reaction-triggered run (see "Reaction wake" below); placeholders `{{PERSONA_NAME}}`/`{{BOT_USER}}`/`{{CHANNEL_ID}}`/`{{CHANNEL_KIND}}`/`{{SELF_ID}}`/`{{REACTED_MESSAGE_ID}}`/`{{REACTED_CONTENT}}`/`{{REACTIONS}}`/`{{LOADED_SKILLS}}`/`{{MEMORY_PATH}}`/`{{CREDENTIALS_PATH}}`/`{{CHANNEL_MEMORY_PATH}}`.
- **`skills/discord/`** — the `discord` skill, in the base `SKILL_NAMES` and so staged for **discord/voice/heartbeat** (heartbeat may deliver to Discord); **mail excludes it** (`skillsExcept("discord")`), mirroring its missing `discord-cli` tool — docs track the tools, and never grant them.

**Response gate** (`classifyMessage`, pure + unit-tested): the **only** hard exclusions are **our own messages** (gated by `selfId` — in the `messageCreate` handler AND defensively in `classifyMessage`, so Baxter never acts on what he posts) and **off-allowlist guilds**. Everyone else — humans **and other bots**, treated identically — resolves to `"respond"` on DM/@mention/reply-to-us, or `"prefilter"` (a pass-through candidate) otherwise. `ChannelDispatcher` debounces per channel, serializes runs within a channel, and caps global concurrency.

**Reaction wake** (`shouldHandleReaction`, pure + unit-tested): a `MessageReactionAdd` on one of Baxter's **own** messages, by **someone other than himself** (the `reactorId === selfId` guard is what stops his own 👀/⏳/✅ status churn from waking him), in an allowed guild, spawns a run via a **separate** `ReactionDispatcher` and a lean dedicated prompt (`discord-reaction-prompt.md`) that strongly biases toward no-op. It's separate from `ChannelDispatcher` (which coalesces per channel, newest-wins) so a reaction can never drop a real message trigger; it debounces per **message** (reusing the base `ChannelDispatcher` machinery via `extends`, overriding only the coalescing to accumulate a message's reactions), under its **own smaller concurrency cap** (`DISCORD_MAX_CONCURRENT_REACTION_RUNS`, default 2) — so total parallel runs are the message cap plus the reaction cap, bounded but not shared. Needs `GuildMessageReactions`/`DirectMessageReactions` intents (non-privileged — no Portal change) and `Message`/`Reaction`/`User` partials (the bot caches almost nothing, so reactions on un-cached messages would otherwise be dropped). Out-of-tokens is ignored for reaction runs — a mere reaction shouldn't trigger a notice.

**Two deliberate changes (2026-07-16, per request):**
- **Haiku pre-filter DISABLED** — the cheap yes/no gate that `"prefilter"`-class messages passed through was removed from `handleChannel`; every non-ignored message now goes straight to a full run. To re-enable, restore `runPreFilter` and gate on `decision === "prefilter"` in `handleChannel` (git history has the Haiku implementation).
- **Other bots treated like humans** — the old bot special-case (bot @mention → strict task-oriented pre-filter; plain/reply bot message ignored unless `DISCORD_TRIGGER_ON_BOTS`) is gone, along with the `DISCORD_TRIGGER_ON_BOTS` flag and the dispatcher's now-dead `fromBot` plumbing. A bot's message triggers Baxter exactly as a human's would; **only his own messages are excluded** (by id). Baxter can't loop on *himself* — self is hard-gated. **A third-party bot loop is a real exposure, and the concurrency caps do NOT bound it**: a ping-pong runs strictly *serially*, so per-channel serialization and the global concurrency cap (which bound *parallelism*) never fire, and `DISCORD_DEBOUNCE_MS` only throttles the rate. The primary code-enforced terminator is now a **per-channel hourly run budget** — `DISCORD_MAX_RUNS_PER_CHANNEL_PER_HOUR` (default 150), enforced in `ChannelDispatcher._enqueue` (over-budget triggers are *dropped*, not queued; `_recordRun` counts each start in a rolling window; per-channel, so one runaway channel doesn't starve others; `0` = unlimited). Behind it, `DISCORD_MAX_SENDS_PER_DAY` (default 2500) remains the coarse daily backstop, with the prompt's "say your piece and stop" as the earlier soft brake. The budget is the code-enforced analog of the removed pre-filter gate. The **same knob also caps the reaction dispatcher**, applied **per channel** (via `ReactionDispatcher._budgetKey`, since that dispatcher's own key is the messageId): reaction runs are no-op-biased and never hit the send cap, so without it nothing code-enforced would bound the run *count* from external reaction churn — and because `shouldHandleReaction` fires on reactions to *any* of Baxter's messages (old ones included), a per-*message* budget would let a reactor cycle across N old messages for N separate budgets, so the budget must key on the channel to bound the whole vector. The budget is enforced at **both** `_enqueue` (drop new over-budget triggers) **and** the `waiting`-queue drain (re-check + drop over-budget waiters): since a reaction's dispatch key (messageId) differs from its budget key (channelId), a burst can park many waiters under one channel budget, and an unconditional drain would run them past the cap. `_overBudget` sweeps its whole timestamp map each call, so it can't grow unbounded across transient keys.

**Context & memory:** the run gets the channel's recent history (sanitized through the same `normalizeTranscriptText`/`neutralizeStructuralMarkers` pipeline as the email transcript) plus **two memory files** — the shared `memory.md` and a per-channel `~/.mail-agent/memory-workspace/discord/<channelId>.md` (`discordChannelMemoryPath`). The prompt tells Baxter to write channel notes liberally and to **author ad-hoc skills** under `LEARNED_SKILLS_DIR` for new bots he learns (not `.claude/skills`, which is read-only to the run — see "Learned skills" above); the daemon stages them into `.claude/skills` each run so they're available next time.

**Send cap:** `send-state.mjs` has a separate Discord counter (`DISCORD_MAX_SENDS_PER_DAY`, default 2500) enforced in `discord-cli`'s `sendMessage` at the actual post (one logical send counts once even if chunked), the flood-guard analog of email's cap.

**Permissions — "anything except manage membership":** enforced twice (Discord doesn't grant it, and `discord-cli` doesn't expose it). Denied: Create Invite, Kick/Ban Members, Manage Roles, Manage Channels, Manage Guild, Administrator, Moderate Members. Everything else granted.

**Setup:** Developer Portal → New Application → Bot tab → enable the **Message Content** privileged intent → copy the token into `app/.env` → OAuth2 URL Generator (scope `bot`, tick the granted permissions above) → open the invite URL → `make discord`.

## Code execution (codapi)

Baxter can write and run **Python/Node code in an offline sandbox** via `code-cli` — a third surface alongside the two browsers, for computation/parsing/data work he can't do by hand. `make codapi` stands up the sandbox as a standing service; all three claude-spawning daemons (`run`/`discord`/`heartbeat`) join its network and grant `Bash(code-cli *)`.

**Why codapi, not Piston/Judge0 (arm64).** This stack runs under Colima's **arm64** VM on macOS. Piston and Judge0 build on `isolate` + prebuilt **amd64** language packages, which don't work here — a spike proved it: official Piston fails at `isolate`'s namespace `clone()` under emulation, and the community `otakulabz/piston-arm` has an arm64 engine but still pulls amd64 runtime packages (`qemu-x86_64: …ld-linux-x86-64.so.2` at exec). **codapi** wins because a *sandbox is a Docker image we choose*, so official arm64 bases (`python:3.12-slim`, `node:20-slim` + libs) run natively. Full rationale + the killed-Piston evidence: `docs/superpowers/specs/2026-07-14-baxter-code-execution-codapi-design.md`.

**Architecture.** `make codapi` builds two sandbox images (`app/sandboxes/{python,node}/Dockerfile`, curated libs baked in — Python `numpy`/`pandas`/`python-dateutil`/`beautifulsoup4`, Node `lodash`/`dayjs`) and the codapi server image (`app/codapi/Dockerfile`: pinned, sha256-verified codapi binary + `app/codapi/codapi.json` + sandbox defs baked in — config is **baked, not bind-mounted**, because under docker-outside-of-docker a bind-mount source must resolve on the host daemon). It runs codapi on `<project>-net` with the **docker socket** (so codapi can launch sandbox siblings) and a `TMPDIR`-matched host bind-mount (`-v $(CODAPI_TMP):$(CODAPI_TMP) -e TMPDIR=$(CODAPI_TMP)`) — codapi writes each run's code to `os.MkdirTemp` and bind-mounts it into the sandbox, and that source path must exist on the host daemon. codapi is **not privileged** (Piston would have been); each run is `docker run --rm --network none --cap-drop all --read-only --user sandbox --memory/--pids-limit` (offline, ephemeral, hardened, capped).

**`code-cli` (`scripts/code-cli.mjs`).** The credential-less scoped boundary, mirroring `mail.mjs`/`discord-cli`: `code-cli python|node`, program on **stdin** or `--file <path>`, raw `fetch` → `http://codapi:1313/v1/exec`. It holds no secret — it just selects the sandbox; the enforced limits are codapi's. **Input-data channel:** with `--file` (program from the file), `code-cli`'s own piped stdin is forwarded to the sandbox as a readable file named `input` (`open("input")` / `readFileSync("input")`) — the only way to get external data into the offline, workspace-blind sandbox, since codapi ignores a request `stdin` field (verified live) but does materialize every `files` entry into the sandbox cwd. Without `--file`, stdin *is* the program (no data channel). The `code` skill (`skills/code/SKILL.md`) is in all three daemons' `SKILL_SRCS` and in `BAKED_SKILL_NAMES` (a learned skill can't shadow it); the shim is installed on PATH by `app/Dockerfile`.

**`files-cli` (`scripts/files-cli.mjs`).** The run's read-only window into its **own workspace** — `files-cli list [subpath]` (sorted files + sizes) and `files-cli grep [-i] <pattern> [subpath]` (fixed-string content search, `file:line: match`). It exists because the run is granted **no** `ls`/`grep`/`find` (bare Bash isn't allowlisted) and Claude Code's native `Grep`/`Glob` aren't granted either, yet the run's durable files (memory, per-channel notes, learned skills, saved scripts, artifacts) all live in `MEMORY_DIR` and it otherwise has no way to enumerate or search them — only the exact paths the prompt injects. **The confinement is the security point, not a UX detail:** unlike native `Grep`/`Read` (which aren't cwd-bounded — the run's `Read` can already open the token files, an accepted residual), `files-cli` resolves every path against `MEMORY_DIR` and refuses anything that escapes it after symlink resolution (and the walk never follows symlinks), so it **cannot** reach the agentmail/discord key files, which live one level up in `~/.mail-agent/` (the *parent* of `memory-workspace/`). So it adds discovery/search over what the run can already read, without widening the reach a filesystem-wide native search would. `grep` is deliberately fixed-string (no regex → no ReDoS on a model-supplied pattern; `code-cli` covers regex); output is capped. No skill — it's documented inline in each prompt's "stay in your working directory" note; the shim is on PATH via `app/Dockerfile`, and all three daemons grant `Bash(files-cli *)`.

**`projects-cli` (`scripts/projects-cli.mjs`).** Cross-cutting **project** notes — one markdown file per project under `PROJECTS_DIR` (`MEMORY_DIR/projects/`), so a project carries context across the email/Discord/heartbeat surfaces (all share `MEMORY_DIR`). Four verbs: `list`, `make <name>` (create-only, `wx` flag → errors on a duplicate slug so notes can't be clobbered), `open <slug>` (prints the whole file), and `save <slug> --expect <version>` (replaces the **whole** file from stdin — a mistyped slug errors with "make it first" rather than upserting a stray project; the prompts/skill tell Baxter to `list` first and to send the full file each save). Writes are **atomic** (temp sibling + `rename`), so a concurrent `open` never sees a half-written file. **Concurrent saves of the same project are guarded by optimistic concurrency (compare-and-swap):** `open`/`make`/`save` vend an 8-hex `version:` token (first 8 hex of sha256 over the file's raw bytes, via the shared `versionToken(buf)`; emitted on **stderr**), and `save --expect <version>` is **rejected** if the file changed since that version — the reject message deliberately echoes only the *stale supplied* token, never the current one (handing back the valid token would let a stale body pass). A brief `proper-lockfile` lock (mirroring `schedule-store.mutate`) wraps the read→compare→rename critical section so two racing saves can't both pass the compare. This closes the old last-write-wins lost-update window (the shared `memory.md`/channel files still have it — they're written with native `Read`/`Write`/`Edit`, no CLI to hang a token on; the prompts now nudge `Edit`-over-`Write` there, and a future `memory-cli` would reuse this exact pattern). Design: `docs/superpowers/specs/2026-07-22-projects-cli-cas-design.md`. On the claude harness a too-large `open` body can be sidestepped for just the token via `open <slug> 2>&1 >/dev/null` (the skill documents it). Names slugify to `[a-z0-9-]` (length-capped, `basename`-defended) and paths resolve under `PROJECTS_DIR` — like `files-cli`, it can't escape the workspace. It's a **shared-core CLI** (`Bash(projects-cli *)` is in `grants.mjs`'s `CORE_TOOLS`, so every surface grants it, and `projects` is in each surface's `SKILL_SRCS` → `BAKED_SKILL_NAMES`); shim on PATH via `app/Dockerfile`; the `projects` skill (`skills/projects/SKILL.md`) carries the full command reference. **Preamble injection:** every run (email, Discord, heartbeat, voice-dispatch) gets the current project list in its prompt via `projectsPreamble()` → a `{{PROJECTS_LIST}}` slot (voice builds the string inline), so a run sees what exists without a `list` call and is nudged to open/update/create. It emits **slug + last-updated date only** — deliberately NOT the free-text title/body, which is indirectly attacker-influenced; slugs are `[a-z0-9-]`-confined and dates numeric, so the injected block can't smuggle an instruction into every future preamble (the title/body a run reads on demand via `open`, never auto-injected). Capped at 40 (the most-recently-updated kept, so an active project isn't buried by the alphabetical head) with an "…and N more" tail; the preamble path passes `listProjects(..., { withTitles: false })` to skip the per-file title read on every render.

**`data-cli` (`scripts/data-cli.mjs` + `scripts/data-sources.mjs`).** A curated read-only gateway to a handful of **preferred data sources** (v1: `espn` scores/schedules, `nominatim` geocoding — both keyless), so Baxter reaches a good structured source instead of scraping the open web, and never handles an API key. `data-cli <source> <path> [--query k=v ...]`, plus `list` (sources + "preferred source per query type" routing hints) and `describe <source>` (base host, key status, and a **pointer** to that source's per-source skill). **The registry owns only the trust-critical bits** (host/auth/key + a one-line routing hint) — deliberately NOT each source's endpoint shape, which is knowledge that rots. That shape lives in a **learned skill Baxter researches, verifies live, and maintains** (`data-cli-<source>`, e.g. `data-cli-espn`): `describe` points him at it — he opens it (via the Skill tool) if it exists, else probes the API and writes it (so the next run just opens it — the learn-once property, but self-healing and verified rather than a stale baked blurb). `describe` does no filesystem lookup; it unconditionally prints the pointer + the write-one-if-missing bootstrap. Adding a source is still one registry entry in `data-sources.mjs`; its shape is Baxter's to fill in. Safe because a learned skill only *suggests* a path — `buildUrl`'s host-lock still confines it, so the security boundary is in the CLI, never the skill. **The host is the security boundary**, exactly as `mail.mjs` fixes the recipient: the registry owns each source's `base` (scheme+host+root path); the run supplies only the path suffix + query values. `buildUrl` builds the URL by **string concat** (never the two-arg `new URL(path, base)` resolution form, which would resolve a leading `/` or `//host` *against* the base and escape it) and then asserts — authoritatively, via the exported `assertConfined` — that the resolved `origin` equals the base's and the pathname is the base path or a child of it (trailing-slash guard blocks a `…/sportsfoo` sibling of `…/sports`); a reject-list (scheme/`//`/leading-`/`/`..`/`%`-encoded traversal/`?`/`#`/backslash/control char on the raw path) is fast-fail hygiene in front of it. So a keyed source's key can only ever be sent to that source's real host — and **keyed sources use `redirect:"manual"`** (a 3xx isn't followed — an open redirect would carry a query-param token cross-origin; manual mode yields an opaque-redirect, so there's no `Location` to surface, the run just retries the canonical endpoint), while keyless sources may follow but re-assert the final origin. **Keys** live in `DATA_KEYS_PATH` (`~/.mail-agent/data-keys.json`, `0600`, keyed by each source's `auth.keyName`, read at runtime) — outside `MEMORY_DIR` like the agentmail/discord key files, so `files-cli` can't enumerate it and the run's env carries none; **same accepted residual** as the tokens (native `Read` by exact path is still possible under the claude harness — onboard only keys whose blast radius fits that). The key value (raw **and** both URL-encoded forms) is **scrubbed** out of all emitted output, and the auth query param is structurally redacted to `[key]` in the reported URL, so an API that echoes the request URL in an error body can't leak it. Responses are **untrusted** (treat like `web-cli`/`WebFetch` content), **capped** with a truncation marker, and time-boxed by a single `AbortController` over fetch + body read (mirrors `web-cli`). Shared-core CLI (`Bash(data-cli *)` in `CORE_TOOLS`; `data` in each surface's `SKILL_SRCS` → `BAKED_SKILL_NAMES`); shim on PATH via `app/Dockerfile`; the `data` skill (`skills/data/SKILL.md`) carries the command reference. Design: `docs/superpowers/specs/2026-07-19-data-cli-design.md`.

**`skills-cli` (`scripts/skills-cli.mjs`).** A **read-only discovery** gateway into the open agent-skills ecosystem (`npx skills` / [skills.sh](https://skills.sh)), so Baxter can **find** skills and **suggest** them to the operator — installing stays a human, host-side action. The **only** verb is `find` (`skills-cli find <query> [--owner <o>] [--limit <n>]`): a host-locked `GET https://skills.sh/api/search` (the endpoint `npx skills find` uses), returning validated metadata rows `{slug, name, installs, owner, repo, url, installCommand, trusted}`. There is **deliberately no `add`/install verb** — so a prompt-injected run can't install through this CLI (the *residual* is the pre-existing web-fetch + unscoped-`Write`→`learned-skills` path, gated by the `skill-discovery` skill's explicit "never fetch/transcribe an ecosystem `SKILL.md`" prohibition + operator audit; the CLI itself adds no install capability). **Every field the operator or Baxter acts on is composed in the CLI from a strict, validated parse of the attacker-influenced registry response — never naive interpolation:** `owner` matches a GitHub-owner charset (`OWNER_RE`, ≤39, start/end alphanumeric — no flag-shaped `--x`), `repo`/`slug` match `SEG_RE` (alphanumeric first char → no leading-`-`, no dot-only `..` root-escape, ≤64); `url` (constructed `github.com/<owner>/<repo>`) and `installCommand` (`npx skills add <owner>/<repo>@<slug>`) are emitted **only** when their parts validate, else `null` + a labeled `sourceRaw` — so a hostile `source`/`id` (`; curl | sh`, backticks, `-g`, `../..`) can never reach a shell command or a flag-shaped argument the operator pastes. `trusted` is an **exact case-insensitive ASCII** match against a verified-vendor allowlist (`vercel-labs`/`vercel`/`anthropics`/`microsoft` — deliberately not `anthropic` singular; env-extensible via `SKILLS_TRUSTED_OWNERS`, validated the same way); **install counts are a weak, self-reported, gameable signal, never the trust signal or the sort key** (rows are ordered trusted-first). Host-locked exactly like `data-cli` (`buildSearchUrl` can't produce an off-origin URL — fixed `/api/search` path, run supplies only encoded query *values*; exported `assertConfined` re-checks origin+path); output is **untrusted** (metadata only, never a skill body), capped + time-boxed. The registry base is a const default overridable only by the **operator** (`SKILLS_REGISTRY_BASE`, validated as a bare `scheme://host[:port]` origin — http/https, no path/query/userinfo) — the run can't set it (env-prefix doesn't match the `Bash(skills-cli *)` grant; `run_cli` is shell-less). Shared-core CLI (`Bash(skills-cli *)` in `CORE_TOOLS`; `skill-discovery` in each surface's `SKILL_SRCS` → `BAKED_SKILL_NAMES`); shim on PATH via `app/Dockerfile`; the `skill-discovery` skill carries the trusted/untrusted "recommend vs just-print-the-command" behavior. Vetted skills are installed **baked** via **`make add-skill SKILL=owner/repo@slug [NAME=…]`** (`scripts/add-skill.mjs`): it fetches with the ecosystem's own `npx skills add` into an ISOLATED temp dir (never the repo), copies the vetted dir into `app/skills/<name>/`, and appends `<name>` to the base `SKILL_NAMES` (shadow-guarded: refuses an existing baked name; `addSkillToGrants` is unit-tested since it's surgery on the security-boundary file), then runs the grants tests. Nothing goes live until the operator reviews the new SKILL.md + `git diff` and rebuilds — it only stages working-tree changes, so the fetch is automated but the vetting stays the operator's. Never into the run-writable `learned-skills/` tier, where vetting wouldn't persist. Remaining fast-follow: daemon-side logging of newly-appeared learned-skill names. Design: `docs/superpowers/specs/2026-07-21-skills-cli-discovery-design.md`.

**Security posture.** Core property is **offline execution** — `network:"none"` on every run, so even a prompt-injected run can only compute and read its own output; no exfil path. The enforced maxima live in `codapi.json` (baked into the server image), because the codapi API is unauthenticated and reachable on the network other ways (e.g. Baxter's browser CLIs can `eval` a `fetch` to it) — so `code-cli` is a scoping layer, not a hard boundary, and the limits must be server-side. **Residual risk:** the codapi container holds the docker socket (host-root-equivalent), so a compromise *of codapi itself* reaches the host; the sandbox hardening (drop-caps/read-only/non-root/offline) stands between an escaped sandbox and that socket. This is the same accepted class as Piston-privileged (widely run for public Discord code bots); mitigate by keeping codapi + the base images updated. Optional later hardening: an authed proxy in front of codapi, or `runtime:"runsc"` (gVisor) per sandbox.

**Artifacts (media out).** Sandbox code can produce files (charts/images/PDF/audio) — Python has Pillow/matplotlib(Agg)/reportlab plus stdlib `wave`. Since the sandbox is read-only except a bounded `/tmp` **tmpfs** (added in `codapi.json`) and files there die with the ephemeral container, artifacts come back **over stdout**: a program writes to `/tmp/artifacts/`, the `emit-artifacts.sh` wrapper (the sandboxes' `run` command) base64-frames each file, and `code-cli` decodes them into `<cwd>/artifacts/`. The frames are delimited by a **random boundary** `code-cli` mints and passes in as `/sandbox/.artifact_boundary` — but that's **collision-avoidance, not authentication**: the sandbox program can read that file and forge frames, so `code-cli` treats every frame as untrusted (basename-sanitizes the name, size-checks, skips a bad one — never escaping `artifacts/` or aborting the run). Per-artifact cap 8 MB (and ~10 MB cumulative per run); `noutput` raised to 16 MB so base64 isn't truncated. `discord-cli send|reply|send-thread --file <path>` uploads a workspace file as a Discord attachment (multipart, ≤25 MB, one send). Email attachments are a deferred fast-follow. See `docs/superpowers/specs/2026-07-15-baxter-sandbox-artifacts-design.md`.

## Heartbeat scheduler (`make heartbeat`)

Baxter can act on a schedule, unprompted. **`make heartbeat`** runs **one dedicated driver** (`scripts/heartbeat.mjs`, a `poll.mjs`-style node loop, same image/tokens/config-volume, on the shared network) that every ~minute fires **due tasks** from a shared `~/.mail-agent/schedule/schedule.json` and delivers the result to Discord or email. Baxter only *edits* the schedule via **`schedule-cli`** (`add`/`cancel`/`list`) from his normal email/Discord runs — the shim + `Bash(schedule-cli *)` grant are on those two daemons **only**; a heartbeat-fired run is deliberately **not** granted `schedule-cli` (a scheduled task can't schedule or cancel tasks). The three surfaces' tool allow-lists differ (mail: mail + schedule-cli; discord: discord + schedule-cli; heartbeat: mail + discord, but *not* schedule-cli) — and rather than three hand-synced `ALLOWED_TOOLS` strings, they're now **composed from a shared core in `scripts/grants.mjs`** (`MAIL_TOOLS`/`DISCORD_TOOLS`/`HEARTBEAT_TOOLS`). **The tool grants stay per-surface (the enforced boundary); the skill docs come from one base list each surface filters.** `grants.mjs` defines a single base `SKILL_NAMES`, and each surface's `*_SKILL_SRCS` is `skillsExcept(...its exclusions...)`: mail excludes `discord`, heartbeat excludes `schedule`, discord/voice exclude nothing. `BAKED_SKILL_NAMES` (the learned-skill shadow-guard floor) is the base list's names (each surface is a subset, so their union is the base). A staged skill *doc* never grants a *tool*: each skill exclusion mirrors a missing tool (heartbeat has neither the `schedule` doc nor `schedule-cli`; mail neither `discord` doc nor `discord-cli`), and the fail-closed `HEARTBEAT_TOOLS` (no `schedule-cli`), not the doc absence, is what actually prevents a scheduled task from scheduling. So the "keep the allow-rule sources in sync" hazard is gone: one tool source of truth per surface, one base skill list, and the guard can't drift from what the daemons stage.

- **`scripts/schedule-store.mjs`** — the store: pure queue helpers (`selectDue`, `applyClaim`→task|null, `applyOnSuccess`/`applyOnFailure`, `resolveNextRun` via `cron-parser` + an `Intl` wall-time-in-zone conversion, `cronMinGapMinutes`) **plus** locked/atomic I/O (`mutate` = `proper-lockfile` + temp-file `rename`; the lock is held only for the brief read-modify-write, never across a fire) and the log helpers. `envInt` fails **closed** on a bad numeric env var.
- **Queue semantics (SQS-style):** due = `next_run_at ≤ now && visible`. Claim sets a **15-min invisibility window** and returns the task or `null` (a concurrent `cancel` wins — the driver fires only a non-null claim). Success removes a one-shot / reschedules a cron; a **hard** failure retries after the window and gives up after `HEARTBEAT_MAX_ATTEMPTS`; an **out-of-tokens** run leaves the claim and pauses the tick (no attempt burned — a global outage isn't the task's fault). Every fire appends one line to `task-log.jsonl` (`completed|failed|gave-up|skipped`).
- **Enforced limits (code, not prompt):** min 1-hour recurrence for `--cron` (validated over the next 100 occurrences, no wall-clock cap; one-shot `--at` any time), `HEARTBEAT_MAX_TASKS`, and a daily `HEARTBEAT_MAX_FIRES_PER_DAY` cap the driver derives from today's (UTC) log lines (durable across restart). These bound the persistence a prompt-injected `schedule-cli add` could otherwise turn into unbounded autonomous runs.
- **Timezone** is the requester's: Baxter sets `--tz` from what they state/known, asking if a clock-time task needs it; `HEARTBEAT_TZ` is only the fallback. The driver writes `DISCORD_TOKEN_PATH` (0600) at startup like `discord-bot.mjs` and strips the token from fired runs' env. Full design: `docs/superpowers/specs/2026-07-15-baxter-heartbeat-scheduler-design.md`.

## Auth

The mail surface authenticates to **AgentMail** with a single **API key** (`AGENTMAIL_API_KEY`) — no OAuth, no consent screen, no token expiry, nothing to renew (the old Gmail path needed a Google Cloud OAuth client and a 7-day refresh token; all of that is gone). Run **`make inbox`** once at setup: it creates-or-shows Baxter's inbox on AgentMail's default `@agentmail.to` domain and prints `AGENTMAIL_INBOX_ID`/`BAXTER_EMAIL` to paste into `app/.env`.

The key is a full-authority credential, so it's kept out of the spawned run's environment. `poll.mjs`/`heartbeat.mjs` write it to a 0600 file (`AGENTMAIL_KEY_PATH`, `~/.mail-agent/agentmail-key.json`) at startup, and `runtime.mjs`'s `runAgent` — the single spawn path all daemons go through — strips `AGENTMAIL_API_KEY` (and `DISCORD_BOT_TOKEN`) from every run's env (`stripRunSecrets`), so a prompt-injected run can't echo or shell-interpolate it. `mail.mjs` reads the key **env-first-then-file**, so a daemon's own call uses the env and a spawned run uses the file. Residual (accepted, identical to the discord/data key files): the run's unscoped native `Read` can still open the 0600 file by exact path. Full design: `docs/superpowers/specs/2026-07-22-agentmail-migration-design.md`.

## Sandbox constraint (important if you touch `poll.mjs`'s claude spawn)

The spawned `claude -p` run's own filesystem sandbox restricts `Write`/`Edit` to its **working directory**, regardless of what `--allowedTools` grants — confirmed by testing, not documented. `/app` isn't persistent storage anyway (only `/home/node`, the config volume, survives container restarts), so the run's `cwd` is set to `MEMORY_PATH`'s own directory (`~/.mail-agent/memory-workspace/`) rather than `APP_DIR`. Consequences:
- `mail.mjs` is invoked by **absolute path** in the run's `Bash` tool (`MAIL_CLI_PATH`), since the relative form only resolves from `APP_DIR`.
- Also confirmed by testing: path-scoped `Write(<path>)`/`Edit(<path>)` `--allowedTools` rules do **not** get approved headlessly in this Claude Code CLI version — only bare, unscoped `Write`/`Edit` do. The isolated `cwd` is what actually bounds the blast radius, not the permission rule. It contains `memory.md` plus a `.playwright/` folder (`poll.mjs`'s `ensurePlaywrightConfig()` writes a `cli.config.json` there before each spawned run so bare `playwright-cli open` defaults to Chromium — see the Playwright-browser note below). A run's unscoped `Write` can rewrite that config, including `browser.launchOptions.executablePath`/`args`, so treat it as a default the code sets, not a control the code enforces — consistent with this project's deliberately minimal, operational guardrails (see Guardrail philosophy below), but worth knowing before assuming the sandbox is a hard boundary on what `playwright-cli` can launch.

## Playwright browser default

`playwright-cli` resolves its default browser from a `.playwright/` workspace folder found by walking up from its own cwd. The spawned run's cwd (`MEMORY_DIR`) never had one, so with no `--browser` flag it silently fell back to the `chrome` channel — which errors, since real Google Chrome ships no Linux arm64 build and this stack runs under Colima's arm64 VM on macOS. `poll.mjs`'s `ensurePlaywrightConfig()` writes `.playwright/cli.config.json` inside `MEMORY_DIR` pointing at the `chromium` channel (already installed by the Dockerfile's `install-browser --with-deps`, no separate install step needed) before every run, so bare `playwright-cli open` just works. It's a default, not an enforced control — see the Sandbox section above.

## The transcript-forgery sanitization pipeline (`transcript.mjs`)

This is the most-reviewed, most-hardened part of the codebase — a long chain of consecutive automated-review rounds (13 fix commits) found and fixed real bugs here (commits from `aad325e` through `b059964`; `git log --oneline aad325e..b059964` for the full story if you need the reasoning behind a specific piece). It was extracted verbatim out of the old `gmail.mjs` into `transcript.mjs` during the AgentMail migration (behavior-preserving; the one deliberate change is that `formatThreadMessage` now takes a provider-neutral normalized message and normalizes `text` itself). Read this before changing any of `TRIGGER_MARKER`, `MESSAGE_SEPARATOR`, `neutralizeStructuralMarkers`, `neutralizeDanglingSeparatorTail`, `normalizeTranscriptText`, or `formatThreadMessage`.

**The problem:** the model needs the full thread transcript (not just the triggering message) for context, but it also needs to know unambiguously which message to actually respond to — and message content is otherwise attacker-influenced (anyone who threads themselves into an allowed conversation). Positional framing ("respond to the last message") turned out to be unreliable — the trigger is chosen from `list-new`'s candidates by `timestamp`, not by transcript position, so a chronologically-later message (e.g. the agent's own prior reply) can legitimately follow the real trigger in the transcript.

**The design that emerged:**
- Every message in the transcript is explicitly formatted; the actual trigger gets a literal `[^ RESPOND TO THIS MESSAGE]` marker appended (not positional inference).
- Content from senders not on `ALLOWED_SENDERS` (checked per-message, not per-thread) is fully redacted — From/Date/Subject/body all replaced with `[redacted]`/`[content omitted]`, not just the body.
- The agent's own past replies are exempted from redaction via a self-applied `baxter-sent` label (applied by `mail.mjs` on every `reply`/`send`; inbox-side metadata an inbound message can't cause to be applied to itself) — **not** via the `From` header (spoofable). This is the unforgeable analog of Gmail's old `SENT` label. Belt-and-suspenders: the adapter also refuses to trust the agent's **own address via the allowlist**, so a forged `From: <own address>` with no `baxter-sent` label is still redacted even if the own address were (mis)listed in `ALLOWED_SENDERS`. (`formatThreadMessage` receives `isOwn`/`isAllowed` as booleans; the adapter computes them.)
- Both the marker and the separator strings are neutralized wherever they appear in real message content (a fixed-point loop, since naive single-pass replacement can reconstruct an intact separator from overlapping occurrences).
- The neutralization runs on **fully composed** message blocks, not individual fields — a body that merely starts with `---\n\n` or ends with `\n\n---` contains no complete separator on its own but combines with the template's own literal `\n\n` (or the *next* block's real separator, inserted afterward by the join) to forge one. Composition-seam bugs like this were the largest single source of review findings.
- The trigger marker itself is protected by a **random UUID placeholder** during sanitization (not a fixed string — fixed placeholders are embeddable by attacker/forwarded content) and substituted for the real marker only after sanitization completes.
- Transcript-text normalization (`normalizeTranscriptText`) folds `\r\n`/`\r`/Unicode line-break-alike characters (`LINE_SEPARATOR`/`PARAGRAPH_SEPARATOR`/`NEXT_LINE`, i.e. U+2028/U+2029/U+0085) to `\n`, and **strips invisible format characters** (`\p{Cf}` — zero-width space/joiners, LRM/RLM and bidi controls, soft hyphen, etc.) entirely, before any of the above runs — every sanitizer matches literal bytes only, and the model reading the transcript isn't a byte-exact splitter, so any character that visually reads as a line break, or that hides inside/reconstructs a structural token, is an equally valid forgery vector. This is the single place the invisible-char strip lives, so both transcript surfaces (the email transcript, built by `mail.mjs`'s `buildThreadOutput` passing each message through `transcript.mjs`'s `formatThreadMessage`; the Discord `clean()` in `discord-bot.mjs`) inherit it — they all call `normalizeTranscriptText` first. `formatThreadMessage` also normalizes `text` itself now (the old `gmail.mjs` normalized the body upstream in its MIME parser, which disappeared with the Gmail path — so a raw CRLF/U+2028 body could otherwise forge a separator).
- **Two sinks, not one**: `thread.body` (fully sanitized inside `mail.mjs`'s `buildThreadOutput`) and the top-level `thread.from`/`thread.subject` JSON fields (deliberately left raw in `mail.mjs`, since `poll.mjs` needs the real `from` for its own allowlist re-check) both eventually reach the prompt. `poll.mjs`'s `renderPrompt` sanitizes `{{FROM}}`/`{{SUBJECT}}` itself, at the point of interpolation — this was the last bug found in the chain, and it's an easy one to reintroduce if a new prompt placeholder gets added without checking whether it needs the same treatment.

**If you add a new field to the prompt template or the transcript**, ask: is this attacker-influenced text? If yes, does it need `normalizeTranscriptText` + `neutralizeStructuralMarkers` before interpolation, and could it combine with adjacent template text or a subsequent join to form a partial/seam forgery?

## Tests

There's a `node:test` suite (`scripts/*.test.mjs` + `scripts/harnesses/*.test.mjs`) — run it with **`node --test`** from `app/` (bare, so it auto-discovers the `harnesses/` subdir; a flat `scripts/*.test.mjs` glob would miss it). The one Python module with logic worth pinning, `invisible_cli.py`, has a standalone assert-based test (no pytest): **`python3 scripts/invisible_cli_test.py`** from `app/` (covers the corrupt-`storage_state` self-heal helpers; the async browser paths are verified live). No extra dependency; `node:test` is built in. Coverage skews toward the pure, load-bearing logic: the message + reaction dispatchers and the `classifyMessage`/`shouldHandleReaction` gates plus the per-channel run budget (`discord-bot.test.mjs`), `discord-cli`'s chunking/flag-parsing, the schedule store's queue + cron math, `send-state` (incl. a subprocess test that the record()-lock survives concurrent sends), the shared tool/skill grants (`grants.test.mjs` — asserts the per-surface allow-list asymmetries and that `BAKED_SKILL_NAMES` stays the union of what the surfaces stage), `code-cli`, the harness adapter (`harnesses/claude.test.mjs` + `runtime.test.mjs`), and — the setup for the next paragraph — the transcript sanitizers + `formatThreadMessage` (`transcript.test.mjs`), the AgentMail adapter's pure cores against an injected fake client (`mail.test.mjs`), and the central credential strip both as a helper and as applied by `runAgent` (`run-env.test.mjs` + a `runtime.test.mjs` case).

**The sanitization pipeline is now directly unit-tested.** `transcript.test.mjs` exercises `normalizeTranscriptText`/`neutralizeStructuralMarkers`/`formatThreadMessage` directly — redaction, the own-message exemption, the trigger marker's placement, the composition-seam and overlapping-separator forgeries, and un-normalized CRLF/U+2028 bodies. The AgentMail migration's refactor to a provider-neutral normalized `formatThreadMessage` is what made that possible (it no longer needs a live provider payload). It's no longer the thin spot it was, but it's still the highest-stakes code here: if you touch it, add real cases to `transcript.test.mjs`, AND replicate the manual verification — build the image, exec a script that imports the real functions from the built `transcript.mjs`/`mail.mjs` (not a hand-copied reimplementation — that's how a bug in the reimplementation itself once slipped through), test against both crafted attack strings and real production thread data, then run `poll.mjs` live for a few cycles before calling it done.

## A sharp edge: typing Unicode escape sequences

Twice in this project's history, typing an exotic Unicode character (or its `\uXXXX` escape sequence) directly into a source-file edit got silently corrupted somewhere in the tool-call transport — once collapsing NUL bytes to invisible spaces (both `Read` and the automated reviewer's own tooling rendered them as blank, masking a real bug for several review rounds), once expanding a typed ` ` escape sequence into the actual raw character, which then broke the JS parser (U+2028/U+2029 are themselves lexical line terminators, so a literal one inside a regex literal terminates it early). **If you need an exotic/control Unicode codepoint in source, use `String.fromCodePoint(0x...)` — plain ASCII digits, no risk.** Verify with `node -e 'for (const ch of require("fs").readFileSync(path,"utf8")) if (ch.codePointAt(0) > 127) console.log(ch.codePointAt(0).toString(16))'` after any edit that should be pure ASCII.

## Guardrail philosophy

Deliberately minimal by design, not an oversight: the container's only credentials are the AgentMail API key and the Discord bot token (no payment info, no other linked accounts), so the persona is free to browse/register accounts/reply without a permission blocklist. The real safety nets are operational and enforced in plain code (not prompt instructions a run could talk itself out of): the sender allowlist (fails closed), the daily send cap, loop-prevention (never processes its own sent mail — identified by the unforgeable `baxter-sent` label, never the spoofable `From`), and the per-cycle email cap. The extensive sanitization work above exists because thread *content* (not the ability to act) turned out to be the actual attack surface once cross-message context was added.
